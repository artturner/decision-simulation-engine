● Now I have a clear picture. Let me summarize what this is and what the path
  forward looks like.

  ---
  What you have: A branching-scenario simulation engine — teachers import JSON
  scenario definitions, learners play through them, and the system records
  choices/reflections. The backend is FastAPI + Postgres, with a Next.js
  frontend stub. Auth is currently a single static X-Admin-Key header; there are
  no user accounts at all. The schema has no users table.

  ---
  What multi-account + gradebooks requires — the two main gaps:

  1. Auth / user accounts

  The current schema needs a users table and ownership wired through everything:

  - scenarios.owner_id → users.id (teacher owns their scenarios)
  - plays.student_id → users.id (student's session, not just a free-text
  learner_label)
  - Replace the static ADMIN_API_KEY with proper JWT-based auth with roles
  (teacher, student, site-admin)

  The fastest path here is to use Supabase for both Postgres and auth. It gives
  you email/password + OAuth (Google Classroom SSO matters for schools),
  row-level security, and a managed Postgres instance — all in one. You'd keep
  your existing SQLAlchemy models and just add Supabase Auth tokens to the
  FastAPI dependency layer.

  2. Gradebook

  This is primarily a new read model on top of what you already have. A
  gradebook is: for a given scenario and teacher, list all students who played 
  it, their completion status, outcome, and reflection responses. The plays +
  events + reflections tables already capture all this data — you just need to
  scope it by owner_id and add student_id to plays.

  ---
  Deployment stack I'd recommend:

  ┌───────────┬────────────────────┬───────────────────────────────────────┐
  │   Layer   │      Service       │                  Why                  │
  ├───────────┼────────────────────┼───────────────────────────────────────┤

  ---
  My recommendation before writing any code: Decide whether students need
  accounts at all. Two models exist:

  - Lightweight (no student accounts): Students play via a shareable link,
  identified only by their name in the reflection form. Teachers see all plays
  on their scenario. Much simpler to build.
  - Full accounts: Students log in, their plays are tied to their identity
  across sessions, gradebook tracks individuals over time.

  The second is significantly more work (student registration flow, session
  management, email verification). For an MVP targeting a single teacher or
  small school, the lightweight model ships faster and covers most gradebook
  needs.

  Which direction fits your use case? That decision shapes where to start.

❯ Under the lightweight plan, will teachers be able to provide class rolls      
  allowing students to choose their own standardized name (to avoid students    
  entering a non-roll name)?                                                    

● Yes, that's straightforward to add under the lightweight model. The pattern
  would be:

  - Teacher defines a class roll (list of names) and associates it with a
  scenario
  - The shareable play link includes the roll context
  - At the start of a play (or reflection), the student picks their name from a
  dropdown instead of a free-text field
  - Non-roll submission is blocked at the API level

  It requires one new table and a small schema change:

  class_rolls (id UUID, owner_id FK→users, name TEXT, student_names JSONB)
  -- student_names: ["Last, First", "Last, First", ...]

  scenarios → add class_roll_id FK → class_rolls (nullable)

  When class_roll_id is set on a scenario, the POST /public/plays/start endpoint
  would require learner_label to be one of the names in the roll (validated
  server-side), and the frontend shows a <select> instead of a text input.

  Two design questions worth deciding now:

  1. Roll scope: Should a roll be reusable across multiple scenarios (e.g.,
  "Period 3, Spring 2026"), or always 1:1 with a scenario? Reusable is more
  useful for teachers with multiple scenarios per class, but adds a management
  step.
  2. Duplicate plays: If a student plays twice, should the second play be
  blocked, allowed silently, or flagged? The gradebook view gets murkier if a
  student has multiple plays under the same name.

  Both are solvable either way — just worth deciding before the schema gets
  built.

● Both can coexist cleanly, and supporting the picker actually clarifies the
  schema. Here's how it works:

  Two entry points:
  - /class/{roll_id} — student picks their name, sees all visible scenarios
  assigned to that roll. Long-lasting; teacher shares once per class.
  - /scenarios/{slug} — direct link to a single scenario, still useful for
  open/public use or sharing individually.

  The key schema change is that the relationship between scenarios and rolls is
  many-to-many, not scenarios.class_roll_id. Visibility lives on the assignment,
  not the scenario, because the same scenario could be visible in one class's
  picker but hidden in another:

  -- replaces scenarios.class_roll_id
  scenario_roll_assignments (
    id UUID,
    scenario_id   FK → scenarios,
    class_roll_id FK → class_rolls,
    visible       BOOLEAN DEFAULT false,
    sort_order    INT,  -- teacher controls display order
    created_at
  )

  When a student enters via /class/{roll_id}:
  1. They pick their name from the roll dropdown
  2. They see all visible = true scenarios for that roll, in sort_order
  3. Choosing a scenario starts a play with learner_label validated against the
  roll
  4. The play records which roll it came from (add class_roll_id to plays) so
  the gradebook can scope correctly

  When entering via a direct /scenarios/{slug} link, the roll context is absent
  — learner_label is free-text and plays.class_roll_id is null. Teachers would
  still see those plays in analytics but they'd be outside the gradebook's
  roll-scoped view.

  The updated plays table gets one new column:

  plays → add class_roll_id FK → class_rolls (nullable)

  Does the picker need its own slug/vanity URL (e.g., /class/period-3-fall), or
  is the roll's UUID fine as the identifier?

  The architecture is: Supabase Auth handles the login flow and issues a signed
  JWT → teacher sends that token to your FastAPI backend → FastAPI verifies the
